# Cloud vs Self-Hosting — Architecture

## Cloud-native

### What architecture is built around cloud services?
<details><summary>Show answer</summary>

**Cloud native** — an architecture designed from the start to take advantage of cloud services, rather than
software merely moved to the cloud unchanged. The cloud changed two things: the **economic model** (subscribe
to a service instead of buying hardware and licensing software) and, on a technical level, **how data systems
are built** — cloud-native is the name for that second shift.

</details>

### What governs cloud-native architecture?
<details><summary>Show answer</summary>

A root move and its two consequences — three clusters:

- **Root:** [building on services, not machines](#what-governs-the-cloud-native-chain) — the cloud-native move itself.
- **Consequence 1:** [storage and compute pull apart](#what-does-the-storage-and-compute-separation-chain-cover).
- **Consequence 2:** [operations is reshaped](#what-does-the-cloud-era-operations-chain-cover).

Trigger: *build on services not machines → storage separates from compute → operations splits in two.*

</details>

### What governs the cloud-native chain?
<details><summary>Show answer</summary>

A derivation chain — each step forces the next:

1. [If you operate the stack yourself, what are your options?](#if-you-operate-the-stack-yourself-what-are-your-options) 
   — operate raw resources yourself, own or rented.
2. [Cloud-native vs a cloud VM?](#what-makes-cloud-native-different-from-a-cloud-vm) 
   — built on services, not raw machines.
3. [What do you gain from a higher-level service?](#what-do-you-gain-from-a-higher-level-service) — capacity +
   failure handled.
4. [How do cloud services stack?](#how-do-cloud-services-stack) — services on services.
5. [How do you choose which layer?](#how-do-you-choose-which-layer-to-build-on) — control vs hassle.

Trigger: *generic resources → IaaS rents them → cloud-native builds on services instead → which hand you capacity
and failure handling → and stack on each other → so you pick a layer by control vs hassle.*

</details>

### If you operate the stack yourself, what are your options?
<details><summary>Show answer</summary>

You operate everything yourself on raw resources:
- a server (real or virtual)
- an operating system
- network connectivity
- storage

Two ways to get that hardware:
- **Own it** — buy and run the machines (on-premises).
- **Rent it** — an IaaS virtual machine in a provider's datacenter; the vendor owns the metal, you install and
  run everything on top.

Either way the responsibility is the same: you assemble and operate the stack. 
The only difference is who owns the hardware — detailed in
[hosting options](1.2.1_Cloud%20vs.%20Self-Hosting_Choosing.md#self-hosted--what-are-the-hosting-options).

</details>

### What makes cloud-native different from a cloud VM?
<details><summary>Show answer</summary>

Not *where* it runs — both are in the cloud — but **what you build on**:

- **Cloud VM (IaaS)** — rented generic hardware. You install, configure, and manage everything on top.
- **Cloud-native** — builds on higher-level **managed services**, where the provider handles capacity, scaling,
  and failure. Examples:
   - managed databases
   - object storage
   - message queues

So the shift is from operating raw machines to consuming services. 
[What that buys you →](#what-do-you-gain-from-a-higher-level-service)

</details>

### What do you gain from a higher-level service?
<details><summary>Show answer</summary>

The provider absorbs the operational work you'd otherwise carry:

- **capacity** — provisioned for you, no hardware planning
- **scaling** — grows and shrinks with demand automatically
- **fault tolerance** — failures handled by the provider, not you
- **maintenance** — patching, upgrades, backups on their side

So you build on a working service instead of assembling and operating one yourself: 
less operational burden, faster to build. 
[These services **stack** on each other](#how-do-cloud-services-stack).

</details>

### How do cloud services stack?
<details><summary>Show answer</summary>

Cloud-native is **layered** — each service builds on the one below:

- object storage (e.g. Amazon S3 - Simple Storage Service) sits near the bottom — raw storage
- a database (e.g. Snowflake) builds on that storage
- tools and services build on the database
- and so on upward

So you get a stack: raw storage → database → tools on top → … The open question that follows:
[which layer should *you* build on?](#how-do-you-choose-which-layer-to-build-on)

</details>

### How do you choose which layer to build on?
<details><summary>Show answer</summary>

The rule: **higher-level services are more specialized.** So:

- if a high-level service matches what you need — use it; far less hassle than assembling it yourself
- if nothing fits — drop to lower-level components and build it yourself; that's then your only option

In one frame: IaaS treats the cloud as a rented computer; cloud-native treats it as a stack of services you build on. 
The deeper you build from raw components, the more **control and effort**; 
the higher you build, the less **hassle** but the more you're locked to one use case.

</details>

### Self-hosted vs cloud-native, by access pattern — examples?
<details><summary>Show answer</summary>

Two axes: **who operates it** (self-hosted vs cloud-native) × **access pattern**:

| Access pattern | Self-hosted                 | Cloud-native                                              |
|----------------|-----------------------------|-----------------------------------------------------------|
| **OLTP**       | MySQL, PostgreSQL, MongoDB  | AWS Aurora, Azure SQL DB Hyperscale, Google Cloud Spanner |
| **OLAP**       | Teradata, ClickHouse, Spark | Snowflake, Google BigQuery, Azure Synapse Analytics       |

</details>

### How is storage durability handled traditionally?
<details><summary>Show answer</summary>

Disk is treated as **durable** — once written, assumed not lost. To survive a single disk failing:

- **RAID** (Redundant Array of Independent Disks) keeps redundant copies across several disks on the **same machine**
- done in hardware or by the OS, and transparent to the application using the filesystem

Baseline assumption: **storage and compute sit on one machine**, and the local disk is trusted. 
The cloud breaks that → [next](#why-cant-a-cloud-vm-treat-local-disk-as-durable).

</details>

### Why can't a cloud VM treat local disk as durable?
<details><summary>Show answer</summary>

A cloud instance's local disk is **ephemeral** — so cloud-native systems treat it as a cache, not long-term storage. 
It becomes inaccessible when:

- the instance **fails**
- the instance is **resized** (swapped for a bigger or smaller one) — that lands on a different physical machine,
  leaving the old disk behind

So durable storage has to live **somewhere other than local disk** → first attempt:
[virtual disks](#what-does-a-virtual-disk-fix-and-what-does-it-cost).

</details>

### What does a virtual disk fix, and what does it cost?
<details><summary>Show answer</summary>

Virtual disks (Amazon EBS — Elastic Block Store, Azure managed disks, GCP persistent disks) are block storage you
can **detach from one instance and attach to another**. Not a physical disk — a separate set of machines
emulating a block device (blocks typically 4 KiB).

- **Fixes:** lets traditional disk-based software run in the cloud — durable storage that survives the instance.
- **Costs:**
   - the emulation adds overhead
   - **every I/O is a network call**, so the app is very sensitive to network glitches

A patch for old software, not a native design → [cloud-native goes further](#how-do-cloud-native-systems-handle-storage).

</details>

### How do cloud-native systems handle storage?
<details><summary>Show answer</summary>

They avoid virtual disks and build on **dedicated storage services tuned to the workload**.

- **Object storage** (e.g. Amazon S3) is built for long-term storage of fairly large files (hundreds of KB to
  several GB).
- But database rows/values are far smaller than that — so cloud databases:
   - keep small values in a **separate service**
   - pack many values into **large blocks** stored in the object store

Using a storage service distinct from the compute machine forces a deeper structural change →
[next](#what-structural-change-does-cloud-native-storage-force).

</details>

### What structural change does cloud-native storage force?
<details><summary>Show answer</summary>

Storage and compute — traditionally on the **same machine** (disk + CPU/RAM together) — become **separated**,
called **"disaggregation."**

- example: object storage (e.g. Amazon S3 — Simple Storage Service) only *stores* files; 
  to analyze that data you run compute **outside** it
- so the disk and the processor no longer live together

That move has two consequences, which branch from here:

- [analysis-time cost](#whats-the-cost-of-analyzing-data-held-in-object-storage) — data must cross the network
- [multitenancy](#what-does-sharing-hardware-across-customers-introduce) — one shared service can serve many tenants

</details>

### What's the cost of analyzing data held in object storage?
<details><summary>Show answer</summary>

Because storage and compute no longer share a machine, the data must **travel over the network** to wherever the
compute runs:

- object storage (e.g. Amazon S3) holds the files
- the analysis code runs elsewhere and pulls the data across the network to work on it

This is the cost side of disaggregation — independent storage is convenient, paid for in **network transfer**.

</details>

### What does sharing hardware across customers introduce?
<details><summary>Show answer</summary>

**Multitenancy** — instead of one machine per customer, several customers' data and computation run on the
**same shared hardware** behind one service.

- **Upside:** better hardware use, easier scaling, easier management for the provider.
- **Cost:** careful engineering so one customer's activity can't hurt another's **performance** or **security** —
  the isolation problem.

</details>

### Who traditionally ran data infrastructure?
<details><summary>Show answer</summary>

**Database administrators (DBAs)** and **system administrators (sysadmins)** — specialized roles owning the data
infrastructure and backend machines.

More recently the trend merged development and operations into one team with shared responsibility — but *what*
that modern role emphasizes is [its own card](#what-does-devopssre-emphasize).

</details>

### What is the job of operations?
<details><summary>Show answer</summary>

Two goals — and they don't change, only the methods do:

- **deliver services reliably** — configure infrastructure, deploy applications
- **keep a stable production environment** — monitor and diagnose anything threatening reliability

Traditionally this meant heavy **per-machine** work:

- capacity planning (watch disk space, add disks before running out)
- provisioning new machines
- moving services between machines
- installing OS patches

The cloud keeps the goals but changes the work → [next](#how-does-a-cloud-api-change-operations-work).

</details>

### How does a cloud API change operations work?
<details><summary>Show answer</summary>

A cloud service exposes an **API that hides the individual machines** behind it. What that changes:

- **metered billing** replaces fixed-size disks — store data without planning capacity ahead, pay for what you use
- services often stay **highly available even when individual machines fail**

So the focus shifts **from individual machines to services** — the reliability goal is unchanged, but the tools
and processes differ → [which DevOps/SRE codifies](#what-does-devopssre-emphasize).

</details>

### What does DevOps/SRE emphasize?
<details><summary>Show answer</summary>

- **DevOps** — a philosophy merging development and operations into one team with shared responsibility for
  backend services and data infrastructure.
- **SRE** (Site Reliability Engineering) — Google's version of that idea.

The reliability goal is unchanged; the practices evolved. Both emphasize:

- **automation** — repeatable processes over manual one-off jobs
- **ephemeral** VMs and services rather than long-running servers
- **frequent** application updates
- **learning from incidents**
- **preserving system knowledge** in the org as people come and go

This philosophy, plus cloud, splits the operations role itself →
[next](#how-did-cloud-change-what-operations-means).

</details>

### How did cloud change what "operations" means?
<details><summary>Show answer</summary>

A **bifurcation** — "operations" now means two different jobs depending on which side you're on:

- **Provider side** — ops teams at infrastructure companies specialize in delivering one reliable service to many
  customers.
- **Customer side** — consumers of those services aim to spend as little time and effort on infrastructure as
  possible.

The customer side still carries real work, and some of it can't be handed off at all — two branches:

- [what the customer still does](#what-does-cloud-operations-still-demand-of-the-customer)
- [what stays in-house regardless](#what-can-never-be-outsourced-to-the-cloud)

</details>

### What does cloud operations still demand of the customer?
<details><summary>Show answer</summary>

**Different work, not less.** The customer still must:

- **choose** the right service for a task
- **integrate** services with each other
- **migrate** between services

The old tasks don't vanish — they transform:

- **capacity planning → financial planning** — metered billing removes capacity limits, but you track what you
  use and why
- **performance optimization → cost optimization** — faster often means cheaper, so tuning targets the bill
- **quotas/limits** (e.g. max concurrent processes) must be known and planned around

And **integration** is a growing challenge — many vendors, ever more services, no standards — so it often takes
significant manual effort.

</details>

### What can never be outsourced to the cloud?
<details><summary>Show answer</summary>

Some operations stay yours no matter what:

- keeping your **application and its libraries secure**
- managing the **interactions between your own services**
- **monitoring the load** on your services
- **tracking down causes** of problems — performance degradation, outages

The cloud changes the *role* of operations — it does not remove the need for it.

</details>
