## Nonfunctional requirements - Reliability

### Fault vs failure?
<details><summary>Show answer</summary>

Same idea at two levels of scope: what stopped working, the part or the whole.

- Fault: one part stops working correctly — a disk dies, a machine crashes, a dependency has an outage.
- Failure (outage): the whole system stops giving the user the required service — it breaks its SLO.

The relationship: one event can be either, depending on where you stand. A dead disk is a failure of that disk,
but only a fault to the larger system around it. If that system has a copy elsewhere, it tolerates the fault and
never reaches failure. So faults are local and survivable; a failure is a fault that the system failed to absorb.

</details>

### What is Fault Tolerance?
<details><summary>Show answer</summary>

A system is **fault-tolerant** if it keeps serving users despite certain faults.

Example:
A file saved on two disks. One disk dies. The system reads the copy from the other disk and keeps working —
the user notices nothing. That's fault tolerance: one part failed, the whole kept serving.

</details>

### Where does one fault take down everything?
<details><summary>Show answer</summary>

A **single point of failure (SPOF)** — a part the system cannot operate without.
There's no backup, so a fault there escalates straight into a whole-system failure.

Example: one load balancer in front of ten healthy app servers. The servers are redundant; the load balancer isn't.
It dies, and all ten become unreachable — one fault, total outage (failure). That single balancer is the SPOF.

</details>

### How do you know fault tolerance actually works?
<details><summary>Show answer</summary>

Break things on purpose. **Fault injection** deliberately triggers faults — e.g. randomly killing processes — so
the fault-tolerance machinery runs constantly instead of sitting untested until a real fault hits. Error-handling
paths that never run are the ones that fail when needed.

**Chaos engineering** is the discipline built around this.

</details>

### What are common hardware faults?
<details><summary>Show answer</summary>

Individual physical components failing:

- **Hard drives** — mechanical failure, the classic case in any large disk fleet.
- **SSDs** — wear plus occasional uncorrectable bit errors.
- **RAID controllers, power supplies, memory modules** — fail less often than disks but still fail.
- **CPUs** — a small fraction have cores that occasionally compute the wrong result, sometimes silently.
- **RAM** — bits corrupt from cosmic rays or defects; even ECC memory hits uncorrectable errors.

Rare on a small system, but at scale they happen often enough to be normal operation.

</details>

### Tolerating hardware faults — cloud vs single node?
<details><summary>Show answer</summary>

Two different levels of defense.

- **Single-node systems** add **hardware redundancy** inside one machine: RAID across disks, dual power supplies,
  hot-swappable CPUs, backup generators. This keeps one machine running for years — but it works best when faults
  are independent, and in practice failures are often correlated (a whole rack or datacenter can go down at once).
- **Cloud systems** care less about any single machine. They tolerate faulty nodes **in software**, spreading
  across machines so the service survives even a full datacenter outage. **Availability zones** mark which
  resources are physically co-located, so you can place redundancy in separate zones.


Tolerating machine loss also helps operations. A single-server system needs planned downtime to reboot for
patches; a multi-node system does a **rolling upgrade** — restart one node at a time, no service interruption.

</details>

### Hardware faults vs software faults?
<details><summary>Show answer</summary>

The key difference is **correlation**, and it drives how much damage each does.

- **Hardware faults** — mostly **independent**. One disk failing says nothing about the next; other components
  usually keep working, at least for a while. Redundancy handles them well.
- **Software faults** — often **highly correlated**. Many nodes run the same code, so they share the same bug —
  it can hit them all at once. Harder to anticipate, and they cause far more system failures than hardware faults.

Redundant hardware doesn't help: identical copies just fail the same way together.

</details>

### Who is to blame for a preventable mistake?
<details><summary>Show answer</summary>

Usually the **organization's priorities**, not the person. Faced with more features vs. more testing, teams
reasonably pick features — so when the predictable mistake happens, blaming the individual misses the real cause.

</details>

### How should you respond to an incident?
<details><summary>Show answer</summary>

Separate two questions and ask them in different rooms.

- **Incident investigation → blameless.** People share full details without fear of punishment, so others can
  learn to prevent it. This may surface a need to shift priorities, fund neglected areas, or fix incentives. Be
  suspicious of simplistic answers — neither "Bob should've been careful" nor "rewrite it in Haskell" helps. Learn
  how the **sociotechnical system** really works from the people in it, and improve it from that.
- **Performance management → separate track.** Blameless is about the incident, not a claim that competence never
  matters. Whether a person is skilled and trying is a real question — handled through review, pairing, and honest
  feedback, and if it points at hiring, at the interview and review process. Just don't run it *during* the
  postmortem: blame at the moment of failure buys silence and kills the learning.

</details>
