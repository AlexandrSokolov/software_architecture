
### What is a distributed system?
<details><summary>Show answer</summary>

A system that involves several machines communicating via a network is called a distributed system.
No shared memory or clock — they cooperate only through the messages they exchange.

</details>

### What is a node?
<details><summary>Show answer</summary>

A process that participates in a distributed system. 

</details>

### Node vs machine?
<details><summary>Show answer</summary>

A node is a process; a machine is the hardware. 
One machine can run several nodes, and a node isn't bound to fixed hardware. 
Conflating them breaks reasoning about the many-nodes-per-machine case.


Note: not everyone is this strict. Many ops docs and diagrams say "node" but mean the machine/VM 
(e.g. Kubernetes calls a worker machine a Node). 
Memorize the process-level meaning; expect the loose one in the wild.

</details>

### Why use a distributed system?
<details><summary>Show answer</summary>

**Forced** — no single-node option exists:
- [Inherent distribution](#when-is-distribution-unavoidable)
- [Cross-service requests](#why-are-microservices-distributed)

**Single-node limits** — one machine can't do the job:
- [Scalability](#outgrowing-a-single-machine)
- [Elasticity](#handling-load-that-varies)
- [Fault tolerance / HA (high availability)](#surviving-a-machine-failure)

**Geography** — where (and when) you run matters:
- [Latency](#serving-global-users-fast)
- [Legal compliance](#when-does-law-force-distribution)

</details>

### When is distribution unavoidable?
<details><summary>Show answer</summary>

**Inherent distribution.** When two or more users interact **with each other**, each on their own device. 
Their devices can only reach each other across a network, so the system is distributed by nature — 
no single-machine version exists.

- Means: chat, multiplayer, collaborative editing — user→network→user. 
- Does *not* mean plain client-server (many browsers, one backend): 
  there users only talk to the server, not each other, so that counts as single-node.

</details>

### Why are microservices distributed?
<details><summary>Show answer</summary>

Because data lives in one service and is processed in another, so it must cross the network between them. 
Splitting work into separate cloud/microservices makes network communication built-in, not optional.

</details>

### Outgrowing a single machine?
<details><summary>Show answer</summary>

**Scalability.** If your data volume or computing requirements grow bigger than a single machine can handle, 
you can potentially spread the load across multiple machines.

</details>

### Handling load that varies?
<details><summary>Show answer</summary>

**Elasticity.** When demand is high at some times and idle at others, a cloud deployment scales up and down to match,
so you pay only for what's in use. One machine must be provisioned for peak and sits wasted otherwise.

</details>

### Surviving a machine failure?
<details><summary>Show answer</summary>

**Fault tolerance / high availability.** When the app must keep working even if a machine, the network, or a whole
datacenter goes down, run on several machines for redundancy. When one fails, another takes over.

</details>

### Serving global users fast?
<details><summary>Show answer</summary>

**Latency.** When users are spread worldwide, put servers in several regions so each user is served from one nearby.
That avoids the wait for packets to travel halfway around the globe and back.

</details>

### When does law force distribution?
<details><summary>Show answer</summary>

**Legal compliance.** Some countries have data residency laws requiring data about their residents to be stored and
processed within the country. Users across several such jurisdictions force data onto servers in each location.

</details>

### Problems with distributed systems?
<details><summary>Show answer</summary>

**The network medium:**
- [In a distributed system, why is any request risky?](#in-a-distributed-system-why-is-any-request-risky)
- [Remote call vs local call — which is faster?](#remote-call-vs-local-call--which-is-faster)

**Operating on it:**
- [Troubleshooting: distributed vs single-node?](#troubleshooting-distributed-vs-single-node)
- [Consistency: distributed vs single-node?](#consistency-distributed-vs-single-node)

For all these reasons, performing a task on a single machine is often much simpler and cheaper than 
setting up a distributed system. When combined with single-node databases such as DuckDB, SQLite, and KùzuDB, 
many workloads can now run on a single node.

</details>

### In a distributed system, why is any request risky?
<details><summary>Show answer</summary>

**Unreliable network.** Every network call may fail — link interrupted, service overloaded or crashed — so a request
can time out with no response. You then don't know if it was received, which makes a blind retry unsafe.

</details>

### Remote call vs local call — which is faster?
<details><summary>Show answer</summary>

**Network is slow.** A call to another machine is far slower than a function call in the same process. 
More nodes aren't always faster — sometimes one machine beats a big cluster, 
and bringing compute to the data beats moving the data.

</details>

### Troubleshooting: distributed vs single-node?
<details><summary>Show answer</summary>

**Hard to troubleshoot.** When the system responds slowly, it's hard to tell which part is at fault, 
since a request crosses many machines. Locating the source of a problem is much harder than on a single machine.

</details>

### Consistency: distributed vs single-node?
<details><summary>Show answer</summary>

**Cross-service consistency.** When each service owns its own database, keeping data consistent across them becomes
the application's problem — no single database enforces it for you.

</details>

### Most common way to distribute a system?
<details><summary>Show answer</summary>

Split into **clients and servers**: clients send requests, servers handle them (usually over HTTP). 
One process can be both. Traditionally called **SOA**, later refined into a **microservices architecture** — 
each service has one purpose, one API called over the network, and one owning team.

</details>

### SOA vs microservices architecture?
<details><summary>Show answer</summary>

Same core idea — build an app as separate services talking over the network. 
Microservices is the refined, stricter form: each service has one well-defined purpose, 
one team that owns it, and typically its own database rather than a shared one.

</details>

### Microservices — advantages?
<details><summary>Show answer</summary>

- Each service is **updated independently**, so teams coordinate less.
- Each service gets the **hardware it needs**.
- The **API hides the implementation**, so owners can change internals without breaking clients.

</details>

### Sharing a database between services?
<details><summary>Show answer</summary>

A shared database:
- makes its schema part of the service's API — so the structure becomes hard to change. 
- lets one service's queries hurt another service's performance. 

Each service owning its own database avoids both.

</details>

### Microservices — costs?
<details><summary>Show answer</summary>

Many services breed their own complexity: 
- testing one means running all it depends on; 
- each needs infra for deploys, scaling, logs, monitoring, alerting; 
- and APIs are hard to evolve — adding/removing fields can break clients, often caught late. 


Each service is simple alone, but you must hold and manage the whole set.

</details>

### When are microservices worth it?
<details><summary>Show answer</summary>

They solve a **people problem** — letting many teams progress without coordinating. Valuable in a large company. 
In a small one with few teams they're unnecessary overhead; build the simplest thing that works instead.

</details>

### What problem does serverless solve?
<details><summary>Show answer</summary>

It removes **machine management**: instead of you starting and stopping instances, the cloud vendor allocates and
frees hardware per request and bills only for actual execution time — metered billing for code, no capacity planning
up front. Catches: execution time limits, restricted runtimes, cold starts, and it still runs on a server.

</details>

### Microservices vs serverless?
<details><summary>Show answer</summary>

No — different questions. 
- Microservices is about **how you split the app** (decomposition/architecture). 
- Serverless is about **who manages the machines** (deployment/operations). 

They're independent: monolith or microservices, each can run serverless or on hand-managed VMs. Don't conflate them.

</details>

### Cloud computing vs supercomputing (HPC)?
<details><summary>Show answer</summary>

HPC - high-performance computing, also known as supercomputing.

Two ways to build large-scale computing. Differences:

| Aspect           | Cloud                                                     | Supercomputing (HPC)                                        |
|------------------|-----------------------------------------------------------|-------------------------------------------------------------|
| Workload         | online services, business data, high availability         | heavy scientific compute (weather, climate, molecular)      |
| Node failure     | keep serving; can't stop the cluster                      | checkpoint to disk; stop, fix node, restart from checkpoint |
| Trust / security | mutually untrusting tenants → isolation, encryption, auth | trusted users; shared memory + RDMA                         |
| Network          | IP/Ethernet, Clos topology                                | specialized meshes/toruses for known patterns               |
| Location         | nodes across regions                                      | nodes assumed close together                                |

Use HPC when: batch, compute-bound scientific work with trusted users — not user-facing services. 
Large-scale analytics can share HPC traits.

</details>
