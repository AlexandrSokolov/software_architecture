## Nonfunctional requirements - Maintainability

### Maintainability — principles?
<details><summary>Show answer</summary>

Every valuable system eventually becomes legacy, so design for the people who maintain it. 


Principles:
- **Operability** — keep it running smoothly. For the *operations team*, day to day.
- **Simplicity** — keep it easy to understand. For *new engineers now*: consistent, well-known patterns; 
  no needless complexity.
- **Evolvability** — keep it easy to change. For *future engineers*: adapt and extend it for use cases nobody
  anticipated as requirements shift.


Recall handle: run it / understand it / change it — present operations, present people, future change.

</details>

### Operability — what supports it?
<details><summary>Show answer</summary>

Good operability makes routine work easy so the ops team can spend time on high-value tasks. 
What the system can do to help, grouped by what each buys them:

- **Visibility** — expose key metrics for monitoring, and support observability into runtime behavior.
- **Understandability** — good docs, a clear operational model ("if I do X, then Y happens"), and predictable
  behavior that minimizes surprises.
- **Automate, but leave an escape hatch** — 
  - sensible defaults that admins can override; 
  - self-healing where it fits, but manual control over system state when needed.
- **No single-machine dependency** — take any one machine down for maintenance while the system keeps running.

Recall handle: see it, understand it, automate-with-override, survive a machine loss.

</details>

### Monitoring vs observability — plus mainstream tools?
<details><summary>Show answer</summary>

Two related jobs:
- **Monitoring** watches predefined key metrics and fires alerts — answers "is it healthy?"
  (known problems you already thought to check). 
- **Observability** lets you infer internal state from external outputs — answers "why is it behaving like this?" 
  (problems you did not predict). Built on three pillars: **metrics, logs, traces**.

Popular tools, mapped to what they cover:

- **Prometheus** — open-source metrics collection + alerting; de facto standard for cloud-native / Kubernetes.
- **Grafana** — dashboards and visualization layer; sits on top of Prometheus and many other sources.
- **Datadog** — commercial all-in-one SaaS: metrics, logs, and traces (APM) in one platform.
- **Splunk** — enterprise log search and analytics; strong for large-scale log data and security.
- **New Relic** — commercial all-in-one APM / full-stack observability SaaS.
- **Dynatrace** — commercial enterprise APM, AI-assisted root-cause analysis.


Also worth knowing: **OpenTelemetry** (vendor-neutral standard for producing metrics/logs/traces, so you avoid
lock-in) and open-source backends like the **Elastic/ELK stack** (logs) and **Jaeger** (traces).


Recall handle: monitoring = is it healthy, observability = why (metrics/logs/traces); 
- one open-source metrics pair (Prometheus + Grafana), 
- three commercial all-in-ones (Datadog, New Relic, Dynatrace), 
- one log giant (Splunk).

</details>

### Complexity-mired project — what's it called?
<details><summary>Show answer</summary>

A **big ball of mud** — a system so tangled by accreted complexity that it's hard to understand and every change
gets slower and costlier.

</details>

### Simplicity — what to reason about?
<details><summary>Show answer</summary>

Complexity is the main enemy of maintainability, so simplicity is a core design goal. 

To reason about:
1. [Why complexity hurts](#why-does-complexity-hurt) — cost and bug risk.
2. [Why simplicity is hard to define](#why-is-simplicity-hard-to-define) — it's subjective, no clean measure.
3. [The best tool for it](#best-tool-against-complexity) — hide detail behind a clean façade.
4. [How that tool is built](#how-are-abstractions-built) — app-level vs general-purpose infrastructure.

Spine: 
- complexity costs money and breeds bugs → 
- aim for simple → 
- but "simple" is fuzzy → 
- hide detail behind a façade → 
- build those façades app-side or on general-purpose primitives.

</details>

### Why does complexity hurt?
<details><summary>Show answer</summary>

- **Cost** — complexity blows budgets and schedules.
- **Bugs on change** — the harder the system is to reason about, the more a change slips: hidden assumptions,
  unintended consequences, and unexpected interactions get overlooked.

So reducing complexity directly improves maintainability.

</details>

### Why is simplicity hard to define?
<details><summary>Show answer</summary>

There is no objective standard — "simple" is subjective. Example: one system hides a complex implementation
behind a simple interface; another has a simple implementation that exposes internals to users. Which is simpler?
No clear answer.

</details>

### Best tool against complexity?
<details><summary>Show answer</summary>

**Abstraction** — hide a lot of implementation detail behind a clean, simple-to-use façade. 
A good one also serves many uses, so reuse beats reimplementing, and a quality fix in the shared component helps every user.

Examples:
- High-level languages hide machine code, CPU registers, and system calls.
- SQL hides on-disk and in-memory data structures, concurrent clients, and crash recovery.

You still rely on machine code — you just don't think about it directly.

</details>

### How are abstractions built?
<details><summary>Show answer</summary>

Two levels:
- **Application-specific** — design patterns and domain-driven design (DDD).
- **General-purpose (infrastructure)** — database transactions, indexes, event logs. 
  These are the book's focus; you implement things like DDD on top of them.

</details>

### Evolvability — what enables change?
<details><summary>Show answer</summary>

Evolvability = agility at the *data-system* level (several apps/services), the term used to keep it distinct from
team-level agility.

What makes a system easy to change:
- **Simplicity and good abstractions** — the ease of adapting a system is tightly linked to these.
- **Loose coupling** — loosely coupled, simple systems change more easily than tightly coupled, complex ones.

Team-level support (borrowed from Agile): **TDD** and **refactoring** as technical practices for a
frequently-changing environment.

Recall handle: same root as simplicity — simple + loosely coupled = easy to change; 
"evolvability" is that idea at system scale.

</details>
