### What determines whether to own or outsource a capability?
<details><summary>Show answer</summary>

Core competency and competitive advantage belong in-house. 
Non-core, routine, or commodity capabilities belong to a vendor — 
the organization that tries to own everything optimizes for the wrong things.

</details>

### How does the own-vs-outsource choice apply to software?
<details><summary>Show answer</summary>

Own-or-outsource splits into two questions about what you're using:

- who **writes** the software
- who **runs** it

The answers form a spectrum with three points:

- **Bespoke** — you write it and you run it. Fully owned, one end.
- **Self-hosted** — you don't write it (ready-made open source or commercial) but you run it yourself. The middle.
- **SaaS / cloud** — a vendor writes it and runs it; you reach it only through a web interface or API. The other end.

Most real choices land in the middle. Example of self-hosted: pull down MySQL and run it on a server you control.

See also: [Self-hosted — what are the hosting options?](#self-hosted--what-are-the-hosting-options)

</details>

### Why is custom software called "bespoke"?
<details><summary>Show answer</summary>

From British tailoring: a **bespoke** suit is cut to your exact measurements, the opposite of off-the-rack. 
The word is the old past participle of **bespeak** — to speak for, to order in advance — so the cloth was "spoken for" 
by one customer before being made. It generalized to anything custom-made to order for a single client.

So bespoke software = written specifically for you, not bought ready-made.

Mnemonic: **bespoke = be-spoke-for = ordered ahead just for you.**

</details>

### Self-hosted — what are the hosting options?
<details><summary>Show answer</summary>

One axis: who owns the hardware you run on.

* On-premises — hardware you control. The term still applies to a rented rack in someone else's datacenter
  (colocation), not only machines physically on your site.
* IaaS (Infrastructure as a Service) — a virtual machine you rent in the cloud. 
  The vendor owns the hardware; you install and run the software on top.

So both are "you operate it yourself" — they differ only in whether the metal underneath is yours or rented.

See also: [How does the own-vs-outsource choice apply to software?](#how-does-the-own-vs-outsource-choice-apply-to-software)

</details>

### What does "on-premises" mean?
<details><summary>Show answer</summary>

*Where* software runs: on hardware in a facility you control — the location axis, not who operates it.

"Premises" = the property a business occupies (old legal usage — the property described earlier in a deed), so
on-prem literally means "on your own property."

Loosened in practice to include a rented rack you control in someone else's datacenter (colocation) — still
"on-prem" because you control the machines, even if not the building.

</details>

### What does using a cloud service actually mean?
<details><summary>Show answer</summary>

Using a cloud service instead of running comparable software yourself means 
you **hand the operation of that software to the cloud provider**. 
They run it; you consume it.

</details>

### Self-hosted vs on-premises?
<details><summary>Show answer</summary>

No — they feel identical but cut on different axes:

- **self-hosted** — *who operates it*: you run ready-made software yourself, keeping control and the operational
  work (vs SaaS, where the vendor runs it).
- **on-premises ("on-prem")** — *where it runs*: a facility you control. "Premises" = the property a business
  occupies (old legal usage — the property described earlier in a deed), so on-prem literally means "on your own
  property." Loosened in practice to include a rented rack you control elsewhere (colocation).

They overlap but aren't synonyms: self-hosted can run on-prem **or** on rented cloud VMs (IaaS). On-prem is a
*location*; self-hosted is an *operating responsibility*.

See also: [Self-hosted — what are the hosting options?](#self-hosted--what-are-the-hosting-options)

</details>

### What governs the self-host vs. cloud choice?
<details><summary>Show answer</summary>

A cloud service hands the *operation* of software to the provider: you trade **control** for **convenience**.
Whether that trade is worth it splits into two groups that answer **different questions**.

**Group 1 — which side wins for me?** Four factors, each able to tip *either way* depending on your situation:

- **Skill** — do you already know how to deploy and run it? See [skill factor](#self-host-vs-cloud--the-skill-factor).
- **Load pattern** — is your machine count steady or spiky? See [load factor](#self-host-vs-cloud--the-load-factor).
- **Tuning** — do you need it tuned to your workload? See [tuning factor](#self-host-vs-cloud--the-tuning-factor).
- **Cost** — what drives the bill at your load? See [cost factor](#self-host-vs-cloud--the-cost-factor).

Each of these can point toward self-hosting **or** toward cloud — that's why they're a weighing.

**Group 2 — what does cloud cost me no matter how Group 1 comes out?** Not a tiebreaker, a one-directional
penalty: it never favors self-hosting, it only counts against cloud.

- **Lost control** — choosing cloud means giving up control of the software. 
  You can't "win" this back by self-hosting; you simply don't pay it. 
  The four factors decide whether cloud's convenience is worth this fixed price. 
  See [what you give up](#what-do-you-give-up-with-a-cloud-service).

Trigger to regenerate: *cloud = trade control for convenience; four situational factors weigh which side wins,
then check whether that win is worth the one fixed loss — control — that picking cloud always costs.*

</details>

### Self-host vs. cloud — the skill factor?
<details><summary>Show answer</summary>

If you **already know** how to deploy and operate the system, self-hosting is often cheaper and no harder.
If you **don't**, a cloud service is usually faster than learning it.

</details>

### Self-host vs. cloud — the load factor?
<details><summary>Show answer</summary>

About **scaling ability**, not the bill:

- **Steady, predictable** load (machine count barely changes) favors self-hosting — nothing to scale.
- **Spiky** load that swings a lot favors cloud — you scale resources up and down as demand moves, instead of
  sizing for a peak that rarely arrives.

The cost consequence of this (idle peak capacity) lands in [the cost factor](#self-host-vs-cloud--the-cost-factor).

</details>

### Self-host vs. cloud — the tuning factor?
<details><summary>Show answer</summary>

The axis: **fit to your workload** vs **breadth of operating experience**. Each side wins a different thing.

- **Self-host → you can tune it.** You control every knob and match the system to your exact workload. 
  Examples: 
  - set a database's cache size and page size to your read/write mix; 
  - pin services to specific CPUs/NUMA nodes for latency-sensitive work; 
  - choose a filesystem and kernel parameters for your I/O pattern.
  
  A managed provider exposes only the knobs they choose to — usually a small subset — 
  and won't make one-off changes for you.

- **Cloud → you inherit their experience.** The provider runs the same service for thousands of customers, 
  so their defaults, failure handling, and upgrade paths are battle-tested. Examples: 
  - sensible replication and backup already wired up; 
  - known-bad configurations avoided; 
  - performance regressions caught across their fleet before they reach you. 
  
  You give up fine control, but the baseline you start from is often better than what you'd tune to alone.

So: tune it yourself for a **perfect fit**, or take the cloud's **proven-good default** and skip the tuning work.

</details>

### Self-host vs. cloud — the cost factor?
<details><summary>Show answer</summary>

Cost isn't a sibling factor — it's **downstream**: skill, load, and capital all feed the bill. Four drivers:

- **Capital** — self-hosting means buying hardware and licenses upfront; cloud converts that to pay-as-you-go.
- **Labor** — running it yourself costs staff to operate it; if you lack the skill, hiring or training is
  expensive (from [the skill factor](#self-host-vs-cloud--the-skill-factor)).
- **Capacity** — steady load is cheap to own; spiky load is cheaper to rent, since owned peak capacity sits idle
  most of the time (from [the load factor](#self-host-vs-cloud--the-load-factor)).
- **Ops** — cloud doesn't remove the ops team; it frees them from basic sysadmin toward higher-level work.
- **Horizon** — rent always includes the provider's margin, so for a **steady, long-lived** load that you'd run
  anyway, owning eventually undercuts renting: the upfront capital amortizes while the rental meter never stops.
  This is why large steady workloads sometimes **repatriate** (move back from cloud to owned hardware) once scale
  and predictability are high. But horizon alone doesn't decide it — it only favors owning when load is also
  **steady**. For **spiky** load, renting can win indefinitely, because owned peak capacity sits idle and that
  waste never goes away. So short vs long term matters, but it pulls the answer only together with load shape,
  not on its own.

Rule of thumb: steady, predictable load → owning is often cheaper; variable load → paying only for what you use wins.

</details>

### You chose cloud — what could change that?
<details><summary>Show answer</summary>

The four factors can all say "cloud," yet two considerations they don't price can still change the decision:

- **Independence / provider power** — a cheaper bill now ignores that the provider can later raise prices, 
  change terms, or cut you off. Renting always includes their margin, so a short-term win can become a long-term loss,
  and you're exposed to their decisions. (This is the cost side of [lost control](#what-do-you-give-up-with-a-cloud-service).)
- **Horizon** — for a **steady, long-lived** load you'd run anyway, owning eventually undercuts renting as the
  upfront capital amortizes while the meter never stops. Pulls toward owning only when load is also steady.

**The middle answer — hybrid + cloud bursting.** Own a baseline fleet sized to your steady load and run the bulk
there; spill **only the spike overflow** to rented cloud capacity. You pay the provider's margin on just the
thin spiky slice, and you stay independent: if they hike prices, you lose only burst headroom, not your service.
During a squeeze you accept degraded peaks (queue, shed load, slow down) and survive on your own hardware.

It isn't free — three costs decide if it's worth it:

- **State gravity** — stateless compute (web servers, batch jobs) bursts easily; **stateful** systems, especially
  databases, don't — you can't spill half a database for an afternoon. Burst suits the compute tier, not the data tier.
- **Standing complexity** — two environments, two deploy paths, networking and sync between them, both tested. It's
  ongoing, not one-time, and needs real operational skill.
- **Egress fees** — providers price moving data *out* high on purpose (a lock-in lever); a design that shuttles
  lots of data both ways can be eaten by egress.

So: cloud can win on the numbers yet lose on **independence and horizon** — and hybrid bursting buys back
independence for **burstable, mostly-stateless** workloads, paid for in complexity and egress.

</details>

### What do you give up with a cloud service?
<details><summary>Show answer</summary>

One root loss — **you don't control the software** — splits into two kinds of consequence:

**Operational — you can't act on the running system:**
- **Missing feature** — you can't add it yourself; you can only ask the vendor and hope.
- **Outage** — you can't dig in; you wait for them to recover.
- **Hard to debug** — no access to OS metrics, internals, or server logs, so bugs and slowdowns are opaque.

**Strategic — you're exposed to the provider and the wider world:**
- **Lock-in** — if they shut down, raise prices, or change the product, you can't keep the old version; 
  you're forced to migrate. Worse when no standard API exists to switch against. Mitigated by
  [hybrid bursting](#you-chose-cloud--what-could-change-that).
- **Geopolitics** — a provider in another country can be cut off by sanctions if relations sour.
- **Data trust** — you must trust them with your data, which complicates privacy and compliance.

Note: even so, cloud and hybrid keep growing — but won't host *everything* (legacy systems, very latency-sensitive
workloads stay in-house). That "what cloud can't fit" question is its own axis.

</details>

### What can't move to the cloud?
<details><summary>Show answer</summary>

A **gate before the trade-off, not a factor in it**: asks whether cloud can do the job *at all*. 
A "no" overrides the four factors — you never reach them.

- **Legacy systems** — built assuming a fixed machine they own (specific OS, hardware dongle, direct disk). 
  Cloud assumes the opposite; rewriting is too costly, so they can't move — not "owning is cheaper," 
  but "moving isn't feasible."
- **Latency-sensitive** — microseconds matter (trading, robotics, telecom). 
  Cloud adds network hops and shared hardware = unpredictable delay; you need the machine next to what it controls.

</details>
